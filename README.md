# 🌐 VPN Gateway: 高可用家庭科学上网方案

## 1. 核心目标
建立一个基于 **阿派 (Local Gateway) + 老谷 (Google VM) + Cloudflare (CDN)** 的三位一体科学上网链路。
*   **低成本**：利用 Google Cloud 免费层及 Cloudflare 免费 CDN。
*   **高隐蔽**：隐藏老谷真实 IP，通过 WS+TLS 协议伪装流量。
*   **省钱策略**：利用 Cloudflare 规避 Google Cloud 昂贵的“直接出站至中国大陆”流量费。
*   **全家无感**：阿派作为热点/网关，实现家庭设备免客户端接入。

## 2. 流量拓扑与“双驱动”架构 (Dual-Engine Architecture)

为了兼顾**极致速度**与**极致稳定性**，本方案采用双链路并行设计：

### 2.1 链路 A：CDN 稳健流 (老谷 + Cloudflare) —— 核心保命链路
*   **路径**：`[终端设备]` -> `[🐾 阿派]` -> `[WebSocket + TLS]` -> `[🟠 Cloudflare]` -> `[👴 老谷]` -> `[🌍 目标互联网]`
*   **协议选型**：VLESS / VMess + WebSocket + TLS。
*   **核心优势**：
    *   **隐藏真实 IP**：利用 Cloudflare CDN 隐藏老谷服务器地址，防封能力极强。
    *   **规避高额流量费**：通过 CF 的内网对等传输，规避从 Google 直接出站到中国大陆的高昂计费。
    *   **作为基座**：即使其他链路被封锁，此链路依然能提供稳定的低速/中速连接。

### 2.2 链路 B：UDP 暴力流 (传家宝 VPS) —— 速度担当链路
*   **路径**：`[终端设备]` -> `[🐾 阿派]` -> `[Hysteria2 (UDP)]` -> `[💾 传家宝 VPS]` -> `[🌍 目标互联网]`
*   **协议选型**：Hysteria2 (Hy2)。
*   **核心优势**：
    *   **榨干带宽**：基于 QUIC/UDP，在丢包率高的弱网环境下依然能爆发出极高带宽。
    *   **低延迟**：适合 4K 视频、大文件下载及实时交互任务。
*   **风险防范**：由于直连 IP 且流量特征明显，仅作为按需开启的提速备线。

## 3. 技术选型与决策背后的思考 (The "Why")

### 3.1 为什么选 Google VM (老谷)?
*   **极致稳定性**：相比于廉价的搬瓦工或 Vultr，GCP 的内网骨干网极其强大。
*   **免费层策略**：利用 `e2-micro` 的免费额度，节省了长期运行的固定月租成本。
*   **纯净 IP**：作为顶级云厂商，其 IP 段虽然常驻 GFW 观察名单，但业务往来多，不容易被整体“一锅端”。

### 3.2 为什么必须垫一层 Cloudflare (CF)?
*   **屏蔽直连费用 (核心原因)**：Google Cloud 直接向中国大陆出站的流量费用极高（约 $0.23/GB）。通过 CF，流量先走 Google 内部到 CF 的 Peering（对等连接），再由 CF 发往用户。由于 CF 到中国大陆通常走公共 Peer，这能显著降低甚至规避直连中国产生的高昂账单。
*   **IP 盾牌**：避免老谷的真实 IP 直接暴露。即便流量被识别，墙封锁的也只是 CF 的边缘节点，我们可以通过更换 CF 解析或开启 Proxy 快速复活，而不需要重置老谷的 VM。
*   **WebSocket + TLS 封装**：将翻墙流量伪装成标准网页交互（HTTPS），不仅兼容性好，且更容易在 CF 这种 CDN 架构中稳定穿透。

### 3.3 为什么本地用“阿派”做热点/网关?
*   **设备无关性**：家里的小米音箱、老旧平板或 Switch 都不支持安装翻墙客户端。阿派作为热点，在硬件层直接把流量“洗白”，所有接入设备无感上网。
*   **智能分流 (v2rayA)**：在阿派上运行 v2rayA 可以精确控制：只有访问 Google 的流量走老谷，访问百度、B站的流量原地直连。这既节省了老谷的流量费，又保证了国内视频不卡顿。

### 3.4 协议深度选型科普 (Protocol Comparison)

针对本方案中的双链路，我们对核心协议进行了如下分工与对比：

| 协议组合 | 流派 | 核心特点 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **VLESS + WS + TLS** | **经典套娃派** | 极其轻量，去除了冗余加密，专门配合 Cloudflare 的 WebSocket 转发。 | **老谷 (GCP) + CF** 链路，追求极致稳定与节流。 |
| **Hysteria2 (Hy2)** | **暴力开路派** | 基于 QUIC (UDP)，在丢包严重的线路上依然能强制跑满带宽。 | **传家宝 VPS** 链路，追求 4K 视频、大文件秒开。 |
| **Reality (黑科技)** | **身份伪装派** | 模拟真实大站（如 Apple）的证书指纹，无需申请 SSL 证书，不可配合 CDN。 | 备选方案，仅用于直连模式下的高隐蔽性需求。 |
| **VMess** | **老牌稳健派** | 传统的加密协议，安全但资源消耗略高于 VLESS。 | 兼容旧版客户端时的备选方案。 |

## 4. 部署路线图 (Roadmap)
- [ ] **阶段 1：老谷服务端部署** (Docker 部署 X-ui 或 v2ray，开启 WS 监听端口)。
- [ ] **阶段 2：域名与 CF 配置** (解析域名至老谷，开启“小云朵”Proxy 模式)。
- [ ] **阶段 3：阿派客户端对接** (配置 v2rayA 导入 CF 节点信息)。
- [ ] **阶段 4：热点与分流优化** (开启透明代理，优化 GeoIP/GeoSite 国内流量分流)。

*   监控 `system.log` 查看连接延迟。
*   定期检查 Google Billing 账单，确认流量路径是否通过 CF 成功降低成本。
